AI 자동 점수 산정 파이프라인

사용자가 Analyze 버튼을 누른 순간부터 각 사진에 0~10점이 매겨져 화면에 반영되기까지 — 레이어를 가로지르는 전체 흐름과, 이미지가 텐서로 바뀌고 모델 출력이 점수로 디코딩되는 내부 수식을 정리합니다.

00한눈에 보는 전체 흐름

점수 산정은 UI → 도메인 유스케이스 → 데이터 리포지토리 → AI 서비스 → 워커 아이솔레이트로 내려가고, 결과는 스트림으로 거슬러 올라옵니다. UI는 ONNX/아이솔레이트를 절대 직접 보지 않습니다.

02① 트리거와 상태 누적

우상단 ✨ Analyze 버튼이 analyzeAll()을 호출합니다. 사진이 없거나 이미 분석 중이면 무시합니다. 결과는 Stream으로 한 장씩 도착하며, 그때마다 누적 맵을 갱신합니다. gallery_controller.dart

final analyze = ref.read(analyzePhotosProvider);
state = state.copyWith(analyzing: true, clearError: true);

final running = Map.of(state.results);
_analyzeSub = analyze(state.photos).listen(
  (result) {
    running[result.photoCacheKey] = result;          // cacheKey 로 색인
    state = state.copyWith(results: Map.unmodifiable(running));  // 한 장 도착 → 갱신
  },
  onError: (e, st) { /* 경고 로깅 + 에러 상태 */ },
  cancelOnError: false,
  onDone: () {
    // 한 장도 나오지 않았다면 "0 results" 힌트로 안내
    state = state.copyWith(analyzing: false);
  },
);
  • 색인 키는 photoCacheKey = 경로::수정시각ms::바이트수. 사진의 내용 동일성을 가리키는 안정 키라, 결과가 정확한 사진에 매칭됩니다.
  • 결과 도착은 입력 순서가 아니라 완료 순서입니다(빠른 사진이 먼저 점수가 뜸).
  • 모델을 바꾸면 selectedModelProvider 리스너가 진행 중 구독을 취소하고 results를 비웁니다 — 화면에 이전 모델 점수가 남지 않습니다.

03② 병렬 처리(바운디드 동시성)

OnnxAiService.analyzeAll은 입력이 5천 장이든 동시 인플라이트 요청을 워커 수만큼으로 제한합니다. 워커가 비는 순간에만 다음 사진을 투입하는 "탑업(top-up)" 방식이라 메모리가 입력 크기와 무관하게 평탄합니다. onnx_ai_service.dart

final concurrency = _pool.workerCount;
void scheduleMore() {
  while (inFlight < concurrency && nextIndex < photos.length) {
    final photo = photos[nextIndex++];
    inFlight++;
    _pool.dispatch(photo).then((r) => controller.add(r), onError: …)
      .whenComplete(() {
        inFlight--;
        if (모두 완료) controller.close();   // 첫 오류가 있으면 addError 후 close
        else scheduleMore();                  // 자리 나면 다음 투입
      });
  }
}
백프레셔 유지가 중요합니다. 이 루프를 Future.wait(photos.map(analyze))로 바꾸면 모든 요청이 한꺼번에 큐에 쌓여 백프레셔가 깨집니다 — CLAUDE.md가 명시적으로 금지하는 패턴입니다.

04③ 워커 풀과 아이솔레이트

WorkerPool은 CPU 코어 수에 맞춰 워커 아이솔레이트를 띄우고, 요청을 라운드로빈으로 분배합니다. worker_pool.dart

워커 수

workerCount = (코어수 − 1).clamp(1, 8)

UI 스레드/컴포지터용으로 한 코어를 비우고, 메모리 관리를 위해 최대 8로 캡.

요청 추적

요청마다 정수 id 부여 → _pending[id] = Completer. 워커가 같은 id로 응답하면 해당 Completer를 완료. 모든 워커가 하나의 ReceivePort를 공유합니다.

각 워커는 시작 시 WorkerInit으로 선택된 ModelConfig 전체(경로·전처리·출력 종류)를 받습니다. 모델을 로드한 뒤 WorkerReady 핸드셰이크를 보내야 풀이 요청을 큐잉합니다. isolate_worker.dart

final model = init.model;
final session = OrtSession.fromBuffer(await File(model.path).readAsBytes(), opts);

// 입력 텐서 이름을 그래프에서 자동 발견 ('input' 하드코딩이 과거 #1 실패 원인)
final inputName = session.inputNames.isNotEmpty ? session.inputNames.first : 'input';

final preprocessor = ImagePreprocessor(config: model.preprocess);
final outputKind   = model.output;

05④ 전처리: 이미지 → 텐서

워커는 사진 바이트를 디코드해 모델이 요구하는 모양·정규화의 Float32 텐서로 만듭니다. preprocessing.dart

1디코드 — img.decodeImage(bytes). 실패(손상/미지원 코덱) 시 null 반환 → 그 사진은 "decode failed"로 처리.

2리사이즈 — inputSize × inputSize(기본 224)로 선형 보간.

3정규화 — 채널 정규화 방식은 모델 설정값:

방식수식대상
imagenet(x − mean[c]) / std[c]
mean=[.485,.456,.406], std=[.229,.224,.225]		MANIQA 등 PyTorch 계열
mobilenetx · 2 − 1  (0..1 → −1..1)NIMA / MobileNet 계열
unitx  (0..1 그대로)일반

x는 image 패키지의 rNormalized 등 0..1 값입니다.

4패킹 — 메모리 레이아웃에 맞춰 배열에 채워 넣습니다.

NCHW (PyTorch)

13224224

채널이 가장 느리게 변하는 축. [R평면 | G평면 | B평면] 순으로 저장.

12242243

채널이 가장 빠르게 변하는 축. 픽셀마다 R,G,B 인터리브.

06⑤ 추론 (OrtSession.run)

전처리 텐서를 ONNX Runtime 입력 텐서로 감싸 실행합니다. 끝나면 입력/출력 텐서를 즉시 release()해 네이티브 메모리 누수를 막습니다. isolate_worker.dart

final inputTensor = OrtValueTensor.createTensorWithDataList(pre.tensor, pre.shape);
final inputs = <String, OrtValue>{inputName: inputTensor};   // 그래프에서 찾은 이름 사용
final outputs = session.run(runOptions, inputs);

final score = decodeScore(outputs, outputKind);

inputTensor.release();
for (final o in outputs) { o?.release(); }

07⑥ 출력 디코딩: 텐서 → 점수

모델 출력을 0~10 품질 점수로 바꾸는 단계입니다. 어떤 방식으로 디코딩할지는 모델 설정(OutputKind)이 선언하므로 런타임 추측이 없습니다. output_decoder.dart

DecodedScore decodeScore(List<OrtValue?> outputs, OutputKind kind) => switch (kind) {
  OutputKind.nimaDistribution => _decodeNima(flat),    // NIMA
  OutputKind.scalarSigmoid    => _decodeScalar(flat),  // MANIQA/MUSIQ/CLIP-IQA
};

NIMA — 10-bin 분포 → 평균 의견 점수(MOS)

출력은 길이 10의 소프트맥스 확률 분포입니다. 각 bin은 1~10점에 대한 확률이므로 가중 평균이 점수가 됩니다.

MOS = Σi=0..9 (i + 1) · pi   ∈ [1, 10]

예시 — 분포가 7~8점에 몰린 경우:

p = [.01,.01,.02,.03,.05,.10,.18,.30,.20,.10]
MOS = 1·.01 + 2·.01 + … + 8·.30 + 9·.20 + 10·.10 ≈ 7.4  →  qualityScore = 7.4

Scalar — 시그모이드 스칼라 ×10

MANIQA 등은 0..1 범위의 단일 실수를 출력합니다. 10을 곱해 척도를 맞춥니다. 추가 출력이 있으면 sharpness/faceCount/blink를 채웁니다.

qualityScore = clamp(flat[0] · 10, 0, 10)
flat[0] = 0.83  →  qualityScore = 8.3
// 선택적: flat[1]→sharpness×10, flat[2]→faceCount(반올림), flat[3]→blink(>0.5)
0점은 "분석 안 됨 / 디코드 실패" 센티넬입니다. 실제 추론 결과는 0에 도달할 수 없습니다(NIMA MOS는 하한 1, 스칼라는 양의 시그모이드 ×10). 그래서 0 = 미분석으로 안전하게 구분됩니다.

08⑦ 결과 반영과 best-shot 연계

워커는 InferenceSuccess로 점수를 돌려주고, 풀은 이를 AnalysisResult로 만들어 해당 요청의 Completer를 완료합니다. worker_pool.dart · analysis_result.dart

AnalysisResult(
  photoCacheKey: …,      // 사진 매칭 키
  qualityScore: …,       // 0..10, 높을수록 좋음 (best-shot 선별 기준)
  sharpnessScore: …,     // 0(흐림)..10(선명)
  faceCount: …,
  hasBlink: …,
  computedAt: DateTime.now(),
)
  • 컨트롤러가 이 결과를 state.results[cacheKey]에 넣으면, 해당 cacheKey를 구독하는 그 사진 타일 하나만 리빌드되어 점수가 표시됩니다(수천 장에서도 전체 리빌드 없음).
  • 이렇게 쌓인 qualityScore가 곧 Select Best Shots(★)의 입력입니다. 상위 분위수 임계값 이상을 자동 선택합니다 — 자세한 내용은 docs/select-behavior.html.

09오류 처리

지점처리
이미지 디코드 실패전처리가 null → 워커가 InferenceError('decode failed') 전송
추론 중 예외try/catch로 잡아 InferenceError(e.toString()) 전송(아이솔레이트 경계 직렬화)
워커 오류 수신풀이 해당 Completer를 StateError로 완료 → 서비스가 첫 오류를 기억
서비스 팬아웃개별 실패는 로깅 후 계속 진행, 모두 끝난 뒤 첫 오류를 스트림에 addError
결과 0개예외 없이 한 장도 안 나오면 컨트롤러가 "모델 입력 이름/모양 확인" 힌트 표시

10핵심 불변식

세션 격리

OrtSession은 스레드 안전하지 않음. 아이솔레이트당 정확히 1개, 생애 내내 공유·이동 금지.

바운디드 동시성

인플라이트 ≤ workerCount. 입력 규모와 무관하게 메모리 평탄.

설정이 진실

전처리·출력 디코딩은 ModelConfig가 선언. 파일명 추측 없음.

캐시 동일성

cacheKey = 경로::mtime::bytes. 결과 매칭의 안정 키.

네이티브 정리

run 직후 입력/출력 텐서 release() — 누수 방지.

완료 순 스트리밍

입력 순서가 아닌 완료 순서로 점수 도착.
